# Limoneo - Estado de i18n

Ultima actualizacion: 2026-03-15

Este documento refleja el estado real de la internacionalizacion tras la ultima pasada. Ya no se centra solo en pendientes: separa claramente lo que esta resuelto, lo que esta parcialmente cubierto y lo que sigue pendiente de migrar.

## 1. Infraestructura vigente

### 1.1 Frontend

- El frontend usa diccionarios por idioma en:
  - `resources/js/i18n/es.json`
  - `resources/js/i18n/en.json`
  - `resources/js/i18n/fr.json`
- El acceso a traducciones se hace con `useI18n()` desde `resources/js/i18n/index.js`.
- El locale activo se comparte por Inertia desde `HandleInertiaRequests`.
- Los idiomas soportados en runtime son:
  - `es`
  - `en`
  - `fr`

### 1.2 Backend

- Laravel ya expone locale activo mediante middleware `SetLocale`.
- La API movil canonica usa ademas `SetApiLocale` para resolver locale stateless por `Accept-Language` y responder `Content-Language`.
- El backend usa `__()` para mensajes visibles en controladores y `FormRequest` localizados.
- Se han creado archivos `lang/` especificos para dominios funcionales auditados:
  - `lang/es/profile.php`, `lang/en/profile.php`, `lang/fr/profile.php`
  - `lang/es/contact.php`, `lang/en/contact.php`, `lang/fr/contact.php`
  - `lang/es/avatar.php`, `lang/en/avatar.php`, `lang/fr/avatar.php`
  - `lang/es/catalog.php`, `lang/en/catalog.php`, `lang/fr/catalog.php`

### 1.3 Default locale

- El locale por defecto del proyecto se ha alineado a espanol en `config/app.php`.
- Esto evita que el backend responda en ingles cuando no existe cookie o sesion de idioma.

- o abrir namespace `admin.*` estable y migrarlo gradualmente

Recomendacion:
- si se mete admin en scope, empezar por Dashboard, Orders, Users y Coupons
- no intentar migrar todo el admin en una sola pasada sin antes fijar estructura de claves

### Mojibake visible en navegador

Nota aparte, fuera del bloque principal de migracion:
- en navegador siguen viendose algunos textos con mojibake en navegacion y login

Alcance observado:
- no se trata solo de falta de i18n, sino de un problema adicional de encoding/copy heredado
- debe revisarse especialmente en componentes de navegacion, auth y cualquier fuente que cargue texto legacy o cadenas ya corruptas en JSON/config

Implicacion:
- aunque una pantalla use `useI18n()`, puede seguir mostrando texto roto si la cadena fuente ya esta mal codificada
- esto requiere una pasada especifica de saneamiento de encoding, no solo migracion de claves

## 4.3 Backend no auditado transversalmente

Aunque perfil/contacto/avatar ya estan cubiertos, no se ha hecho una barrida completa de todos los mensajes backend del proyecto.

Sigue pendiente revisar si hay respuestas o validaciones visibles en:
- checkout
- pedidos
- admin
- catalogo y resenas

Objetivo:
- evitar que el frontend este en `fr` o `en` mientras el backend devuelve errores en espanol o ingles por defecto en flujos no auditados aun

## 5. Limpieza pendiente de diccionarios

Ademas de traducir, falta una pasada de mantenimiento.

Pendiente:
- eliminar claves obsoletas no usadas
- consolidar namespaces antiguos y nuevos
- revisar coherencia de naming entre `profile.*`, `static.*`, `dashboard.*`, `special.*`, `catalog.*` y `marketing.*`
- revisar copy legacy que sigue existiendo en JSON pero ya no coincide con el tono actual del producto
- mantener encoding consistente y evitar mojibake en texto heredado

## 6. QA funcional pendiente

Antes de dar i18n por cerrado a nivel de storefront, falta validar manualmente:

1. cambio real de idioma en `es`, `en` y `fr` sobre paginas publicas principales
2. ausencia de keys visibles tipo `foo.bar.baz`
3. consistencia de moneda, fechas y formatos por locale
4. textos largos en mobile para evitar overflow o cortes
5. mezcla accidental de idiomas en header, footer, product page, checkout y paginas especiales
6. mensajes backend en locale correcto cuando hay errores de formulario o acciones fallidas
7. ausencia de mojibake visible en navegacion, login y zonas compartidas

## 7. Orden recomendado desde aqui

1. migrar `ProductPageLayout`, `ProductDetails` y `Checkout`
2. migrar `Footer`, `TopBanner`, `site.js` y `Cookies.jsx`
3. sanear mojibake visible en navegacion/login y revisar fuentes legacy de texto
4. decidir alcance real del admin
5. revisar mensajes backend de checkout/pedidos/admin
6. limpiar diccionarios y ejecutar QA transversal por idioma

## 8. Definition of done minima

Se podra considerar i18n razonablemente cerrado cuando:

- ninguna pagina publica principal muestre texto hardcodeado fuera del sistema i18n
- no queden `alert()` o mensajes de error visibles sin traducir en flujos principales
- frontend y backend respondan en el mismo locale activo
- no aparezcan claves sin resolver en UI
- el storefront pueda recorrerse en `es`, `en` y `fr` sin mezcla evidente de idiomas
- no quede mojibake visible en navegacion, login o superficies compartidas
- los textos globales de navegacion, checkout y product page salgan del sistema de traduccion y no de config fija
